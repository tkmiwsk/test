## 3. 共通・横断的な未定義事項

| 項目 | 論点 |
|------|------|
| **プラットフォーム** | iOS / Android / Webどれを対象にする？（UIの細部に影響） |
| **ナビゲーションの制御** | 未ログイン・未連携状態でどの画面にアクセス制限する？ |
| **通知の種類** | プッシュ通知の種類・内容・頻度の仕様 |
| **キャラのテーマカラー連動** | アクセントカラーが切り替わる対象コンポーネントの定義 |
| **エラー画面** | ネットワークエラー・API障害時の共通エラーUI |
| **オンボーディング完了の定義** | 診断 → 結果 → API連携のどこまで完了したら「設定完了」？ |
| **データ更新タイミング** | SNSデータをいつ/どの頻度で取得・更新する？ |

## 4. 優先度の整理（デザイン着手順）

デザインを進める前に、以下を決める順番を提案します。

### STEP 1（最優先・デザイン開始に必須）
1. プラットフォームを決める（iOS / Android / Web）
2. 診断のロジック（回答→キャラのマッピングルール）
3. 「あとで連携」選択後のフロー

### STEP 2（デザイン中に決める）
4. 各画面の空・ローディング・エラー状態
5. キャラ切替演出の全6パターンのセリフ
6. SNS別の表示指標の差異

### STEP 3（デザイン後に決めてもよい）
7. データ更新頻度
8. 通知の種類・頻度
9. シェア機能の技術実装方式
